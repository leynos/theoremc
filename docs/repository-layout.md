This document is the canonical guide to theoremc's repository structure. It
explains where source code, tests, fixtures, generated artefacts, plans, and
long-lived reference documents belong.

## Path responsibilities

- `AGENTS.md` defines repository-local agent and contributor instructions. It
  supersedes generic instructions for this repository.
- `Cargo.toml` is the workspace manifest. It declares the root `theoremc`
  facade crate, workspace members, dependencies, test targets, and workspace
  lint policy.
- `Makefile` is the preferred command surface for formatting, linting, tests,
  Markdown validation, and other quality gates.
- `build.rs` is the Cargo build-script entrypoint. It discovers theorem files,
  emits rerun metadata, and writes generated suite wiring into Cargo's output
  directory.
- `src/` contains the root `theoremc` crate. The crate re-exports core schema
  and mangling APIs, the `theorem_file!` proc macro, and build integration
  helpers used by the build script and tests.
- `crates/test-helpers/` contains shared test utilities that need to be reused
  across crates, especially helpers for process-global state.
- `crates/theoremc-core/` contains core theorem semantics: schema loading,
  validation, diagnostics, action and harness mangling, theorem-file loading,
  and collision detection.
- `crates/theoremc-macros/` contains the `theorem_file!` procedural macro and
  compile-time expansion tests.
- `tests/` contains workspace-level behavioural and integration tests. Shared
  helpers belong in `tests/common/`, Gherkin feature files in
  `tests/features/`, theorem input examples in `tests/fixtures/`, and snapshot
  output in `tests/snapshots/`.
- `tests/theorem_file_macro_bdd/` contains support code for macro behavioural
  tests that need fixture crates or Cargo/Kani command wrappers.
- `docs/` contains long-lived project documentation. Use
  `docs/contents.md` as the documentation index and this file as the canonical
  repository layout guide.
- `docs/execplans/` contains living execution plans for roadmap tasks. Plans
  must be updated as implementation facts change.
- `target/` is generated by Cargo and must not be edited or documented as a
  source of truth.

## Generated artefacts and fixtures

- Cargo-generated build products belong under `target/` or Cargo-provided
  output directories such as `OUT_DIR`; they should not be committed.
- Theorem fixtures under `tests/fixtures/` and
  `crates/theoremc-core/tests/fixtures/` are committed test inputs and should
  be updated with the tests that depend on them.
- Snapshot files under `tests/snapshots/` and crate-local `snapshots/`
  directories are committed expected outputs. Update them only when the
  user-facing diagnostic or expansion contract intentionally changes.
- Build-script generated theorem-suite files are transient outputs. The source
  of truth is the renderer in `src/build_suite.rs` and its tests.

## Documentation locations

- User-facing behaviour belongs in `docs/users-guide.md`.
- Maintainer workflows and internal conventions belong in
  `docs/developers-guide.md`.
- System architecture and rationale belong in `docs/theoremc-design.md`.
- Accepted architectural decisions belong in ADR files named
  `docs/adr-NNN-short-description.md`.
- Proposed technical changes that need review before acceptance belong under
  `docs/rfcs/` when that directory is introduced.
- Repository structure belongs here, not in the developer's guide.
